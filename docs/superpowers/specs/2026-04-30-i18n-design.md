# Health Advisor i18n 设计规范

## 概述

为 Health Advisor 添加国际化支持，从纯中文扩展为中英双语。核心挑战不仅在前端 UI 文案，还包括 LLM 输出内容的语言约束、Agent Prompt 模板的多语言管理、以及沙箱数据的全面双语化。

## 关键决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 语言切换方式 | 前端手动切换 + localStorage | 简洁可控，用户主动选择 |
| LLM 输出控制 | 混合方案（结构化指令英文 + 沟通风格双语） | 平衡 LLM 稳定性与维护成本 |
| 沙箱数据 | 全面双语 | 完全可控，无遗漏风险 |
| 语言传递方式 | HTTP Header（X-Lang） | 简洁一致，无 URL 侵入 |
| 响应格式 | 单语言响应，结构不变 | 前端无感知，无需语言相关逻辑 |
| 前端 i18n 框架 | next-intl | 原生 App Router 支持，类型安全 |

## 语言标识

```typescript
// packages/shared/src/types/locale.ts
export type Locale = 'zh' | 'en';
export const DEFAULT_LOCALE: Locale = 'zh';
export const SUPPORTED_LOCALES: readonly Locale[] = ['zh', 'en'] as const;
```

## 1. 整体架构与数据流

### 语言传递链路

```
浏览器 (localStorage 'lang' → 'zh' | 'en')
  → 前端手动切换，更新 Context + localStorage
  → API 请求自动注入 Header: X-Lang: zh|en
    → Next.js Middleware: 设置 NextIntl 上下文
    → Fastify LangPlugin: 解析 Header → request.lang
      → Agent Core: prompt 双层模板按 lang 拼接
      → Sandbox: 按 lang 展平双语数据
        → LLM: 输出目标语言内容
        → Fallback: 按 lang 选择预置响应
```

### 核心原则

1. **单一语言贯穿**：一次请求中，语言参数从浏览器到 LLM 全链路一致，不做中间转换
2. **语言作为参数，不是分支**：避免代码中到处出现 `if (lang === 'en')` 的条件分支，通过模板和数据结构消化差异
3. **前端无感知**：API 响应结构不变，内容自然跟随语言，前端不需要做语言相关的逻辑判断

## 2. 前端 i18n 架构

### next-intl 集成

使用无路由前缀模式，通过 React Context 管理语言状态：

```
apps/web/
├── src/
│   ├── i18n/
│   │   ├── request.ts          # next-intl 初始化
│   │   └── navigation.ts       # 无路由前缀配置
│   ├── messages/
│   │   ├── zh.json             # 中文翻译字典
│   │   └── en.json             # 英文翻译字典
│   ├── middleware.ts            # 仅设置 cookie，不处理路由前缀
│   └── app/
│       └── layout.tsx          # NextIntlClientProvider 包裹
├── components/
│   └── language-switcher.tsx   # 语言切换按钮
```

### 翻译字典结构

按功能模块划分命名空间：

```json
{
  "common": { "appName": "...", "loading": "...", "error": "..." },
  "homepage": { "morningReport": "...", "todaySummary": "..." },
  "dataCenter": { "title": "...", "hrvTrend": "..." },
  "advisorChat": { "placeholder": "...", "send": "..." },
  "godMode": { "title": "...", "scenarioInject": "..." }
}
```

### 语言切换组件

位于顶部导航栏右侧，样式 `[中 | En]`。点击后更新 localStorage + React Context，所有组件自动重新渲染，不刷新页面，不改变 URL。

### API 请求封装

统一 fetch 工具函数，自动携带 X-Lang Header：

```typescript
// apps/web/src/lib/fetch-with-locale.ts
async function fetchWithLocale(url: string, options?: RequestInit) {
  const locale = useLocale();
  return fetch(url, {
    ...options,
    headers: { ...options?.headers, 'X-Lang': locale },
  });
}
```

react-query 的 queryFn 统一使用此函数。

### 前端文本处理范围

| 类型 | 示例 | 处理方式 |
|------|------|----------|
| 静态 UI 文案 | 按钮、标题、placeholder | `t('key')` |
| 格式化文本 | "昨天 08:30" | next-intl 日期/数字格式化 |
| 图表标签 | 坐标轴文字、图例 | props 传入翻译后的字符串 |
| God Mode 面板 | 表单标签、选项 | `t('key')` |
| LLM 返回内容 | summary、microTips | 后端已处理，前端直接渲染 |

## 3. 后端语言处理与 Agent Prompt 系统

### Fastify 语言插件

```typescript
// apps/agent-api/src/plugins/lang-plugin.ts
// 解析优先级：X-Lang Header > Accept-Language > 默认 'zh'
// 校验：仅接受 'zh' | 'en'，非法值回退到 'zh'
// 注入：request.lang
```

### Prompt 双层模板结构

```
data/sandbox/prompts/
├── system.md              # 纯结构性指令，固定英文（不变）
├── homepage/
│   ├── template.md        # 语言无关的任务结构定义（英文）
│   └── style/
│       ├── zh.md          # 中文风格指令（语气、措辞、格式要求）
│       └── en.md          # 英文风格指令
├── view-summary/
│   ├── template.md
│   └── style/{zh,en}.md
└── advisor-chat/
    ├── template.md
    └── style/{zh,en}.md
```

### Prompt 组装公式

```
最终 Prompt = system.md + template.md + style/{lang}.md + 数据上下文
```

各层职责：
- `system.md` — 角色定义、安全规则、输出协议（英文，不变）
- `template.md` — 任务结构、数据插值占位符（英文，稳定）
- `style/{lang}.md` — 语气、措辞风格、输出语言指令（按语言切换）

`style/{lang}.md` 中包含硬性语言指令：

```
You MUST respond entirely in {languageName}.
All summary text, microTips, and source descriptions must be in {languageName}.
```

### Fallback 降级响应

改造为顶层双语结构：

```json
{
  "zh": {
    "summary": "你的晨间健康概览...",
    "microTips": ["..."],
    "source": "综合分析",
    "statusColor": "good"
  },
  "en": {
    "summary": "Your morning health overview...",
    "microTips": ["..."],
    "source": "Comprehensive analysis",
    "statusColor": "good"
  }
}
```

按 `request.lang` 选择对应语言内容。

### Agent Core 变更模块

| 模块 | 变更 |
|------|------|
| `executor/` | 创建 Agent 时传入 `locale` 参数 |
| `prompts/` | prompt 组装逻辑改为双层模板拼接 |
| `context/` | 数据上下文中的文本字段按语言选择 |
| `fallback/` | 按 locale 选择降级响应 |
| `routing/` | 任务路由增加 locale 参数传递 |
| `output/` | 无变更（结构不变，只验证格式） |

## 4. 沙箱数据双语结构

### Profile 文件

数值保持原样，所有面向用户的文本字段统一为 `{ zh, en }` 结构：

```json
{
  "id": "profile-a",
  "name": { "zh": "林巅峰", "en": "Lin Dianfeng" },
  "bio": {
    "zh": "32岁男性，互联网公司技术总监...",
    "en": "32-year-old male, tech director..."
  },
  "baseline": {
    "restingHeartRate": 62,
    "fitnessLevel": { "zh": "中等偏上", "en": "Above average" }
  }
}
```

### History 与 Timeline Scripts

纯数值字段不变，描述性字段提供双语。

### Sandbox Package 加载逻辑

增加语言参数，内部展平双语字段：

```typescript
loadProfile(id: string, locale: Locale): LocalizedProfile
```

`LocalizedProfile` 将所有 `{ zh, en }` 字段展平为当前语言的纯字符串。下游消费者拿到的数据结构与改造前完全一致。

### 数据生成脚本

`generate-history.ts` 和 `generate-timeline-script.ts` 更新为同时生成中英文文本字段，保持 seeded PRNG 的确定性。

## 5. 测试策略

### 测试分层

| 层级 | 内容 | 方法 |
|------|------|------|
| 单元测试 | Locale 类型校验、sandbox 加载器展平、prompt 拼接 | Vitest，覆盖 zh/en |

## 6. 实施范围

### 受影响的包

| 包 | 变更性质 | 复杂度 |
|----|----------|--------|
| `packages/shared` | 新增 Locale 类型和工具函数 | 低 |
| `packages/sandbox` | 数据结构改为双语，加载器增加展平 | 中 |
| `packages/agent-core` | prompt 双层模板、fallback 双语、locale 传递 | 高 |
| `apps/agent-api` | LangPlugin、路由层传递 locale | 中 |
| `apps/web` | next-intl 集成、翻译字典、切换组件、API 封装 | 高 |
| `packages/charts` | 图表标签通过 props 接收翻译文本 | 低 |
| `packages/ui` | 通用组件硬编码文案提取 | 低 |
| `data/sandbox/` | 所有 JSON/MD 文件添加双语内容 | 中 |

### 实施阶段

```
阶段 1 — 基础设施
  shared.Locale 类型 → sandbox 双语数据结构 → sandbox 加载器展平

阶段 2 — 后端语言链路
  agent-api LangPlugin → agent-core locale 传递 → prompt 双层模板 → fallback 双语

阶段 3 — 前端 i18n
  next-intl 集成 → 翻译字典编写 → 语言切换组件 → API fetch 封装

阶段 4 — 测试与数据
  language_match Scorer → eval case 双语回归 → E2E 测试 → 数据生成脚本更新
```

### 不做的事情

- 不做 URL 路径前缀（如 `/en/dashboard`）
- 不做浏览器语言自动检测
- 不做语言包懒加载
- 不改 `AgentResponseEnvelope` 的 TypeScript 类型
- 不引入 ICU MessageFormat
- 不考虑第三语言扩展性
